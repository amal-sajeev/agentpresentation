Slide 03 Guardrails & Provability ~140 words · ~1 min

Here's the situation in a regulated bank: an answer you can't verify is a liability. Every claim needs to link to something real, and the system needs to prove it got there through a defensible process.

Four mechanisms do that here. Grounded citations — every claim in every output links to a numbered source. The report doesn't assert things; it cites them. Typed verdicts — the evaluator returns a schema-checked grade. Malformed? Rejected. No exceptions. Runtime guardrails via Model Armor — PII and jailbreak attempts get filtered on every live call, before they touch a model. And deterministic exit — the critic loop is hard-capped at five iterations. The system cannot spin.

Stack those four together. You've got an answer a compliance officer can pick up and walk into a meeting with. That's what this slide is about.

Slide 04 Memory Architecture ~140 words · ~1 min

Every experienced professional keeps files. Not because they're forgetful — because context compounds. What the client told you last quarter matters this quarter. What worked on a similar engagement is worth surfacing before you start from scratch.

This system has three tiers of memory, all behind a single API surface. Short-term: in-session state shared across every agent during a single run. The whole pipeline sees the same context simultaneously. Semantic: Vertex AI Vector Search. Ask about fintech in Southeast Asia, and it pulls similar past engagements by meaning, not keyword. That's recall that scales. Long-term: a per-client Memory Bank on Vertex AI that persists across sessions. Your client's history doesn't reset every call.

Three tiers. One interface. The agents don't manage backends — they just call the surface. Memory done right means the system gets smarter with use.

Slide 05 NLP Routing · Agent Garden ~143 words · ~1 min

The routing layer's job is straightforward: take a plain-English prompt and figure out what it's actually asking for. No special syntax. No templates. The user types, the classifier reads it, and a Python gate decides which pipelines run.

"Sell to Microsoft" — classifier returns a typed verdict, the sales pipeline activates. "Research the market" — sales branch skips entirely. Zero tokens spent on a branch that wasn't needed. That's the conditional logic working exactly as designed.

The Critic-Worker loop sits inside every pipeline. The worker drafts on flash-lite — the cheap model. The critic grades it on flash. Fail? Back to the worker, up to five times. Pass? Forward. The answer that makes it out has been earned.

And the Agent Garden means new capabilities slot in without rewriting the orchestrator. Drop it in, register it, done.

Slide 06 Architecture Overview ~136 words · ~1 min

Let me show you the system end to end. A user prompt comes in. The classifier — the user_input_analyzer — reads it and routes it. The Intelligence Chancellor, a SequentialAgent, picks it up and runs five expert pipelines in declared order: Market Context, Segmentation, Org Intelligence, Sales conditionally, and Prospects.

Every pipeline writes its output to one place — MongoDB. One shared source of truth. Your downstream tools, whatever they are, subscribe to that store. They don't pull from five different systems. They pull from one.

That's the full architecture. Clean input, clean orchestration, clean output. No data scattered across siloed services.

Five experts. One orchestrator. One truth. The complexity lives inside the pipelines, not in how they communicate — and that's precisely what makes the system maintainable at scale.

Slide 07 Cost & Token Strategy ~140 words · ~1 min

The finance team will ask one question: what does this cost per run? Here's the answer, broken into four parts.

About ninety percent of calls run on flash-lite — the cheaper tier. The only calls on the stronger flash model are the critics, because judgment costs more than drafting. That's the right tradeoff.

Critic loops are hard-capped at five iterations. The system won't run up an unlimited tab trying to get an answer right. It tries, grades, passes or stops. Deterministic. The sales pipeline only runs when the prompt targets a specific company — on a pure research run, that branch skips, eliminating roughly seventy-three percent of that branch's token spend. And each agent gets a compact JSON context scoped to what it actually needs. No padding, no bloat.

Four levers. All of them pull in the same direction.

Slide 08 Evaluation Framework ~142 words · ~1 min

Three rings of quality control. Let me walk you through them.

Ring one: the in-loop critic. This runs during every pipeline, inside the run itself. The worker drafts. The critic — a stronger Gemini — grades it against a typed schema. Fail, and it goes back to the worker. Pass, and the pipeline moves forward. Every run. Every time. No exceptions.

Ring two: offline benchmarking. Ragas and DP Val score a golden dataset — market sizing, ICP personas, sector research — across environments. You catch model drift before it affects a client.

Ring three: Model Armor at runtime. Every live call gets filtered for PII and jailbreak attempts. Clean requests go through. Everything else doesn't.

Three rings, running simultaneously. Two of them you'll never notice working, which means they're doing their job. That's a quality framework worth calling quality.

Slide 09 Auditing & Governance ~143 words · ~1 min

Your compliance team will have a list of questions. When did the system make that decision? What inputs did it see? Can we replay the run from six months ago? Can we pull a report on all evaluator failures for Q3?

The answer to all of those is yes. Every agent emits spans through the after_agent_callback hook. OpenTelemetry picks them up — built into ADK, not added later. Those spans export to Langfuse, where you get a full trace waterfall: every tool call, every critic score, every token cost, every latency figure.

Run replay is available for any historical trace. Regression alerts surface rising fail-rates and latency spikes before they reach production. The audit trail is queryable and permanent.

The compliance story here isn't a feature bolted on. It's the architecture. Everything gets logged because everything flows through the same trace pipeline.
